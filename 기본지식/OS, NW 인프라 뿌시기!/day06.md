# Day 6: Load Test (로드 테스트)

## 0. 목표

- 서버가 동시에 여러 요청을 받을 때 어떻게 동작하는지 이해한다
- 시스템의 한계 지점을 찾는다
- 병목이 어디서 발생하는지 분석한다

---

## 1. 전체 흐름 복습 (Day 1 ~ 5 연결)

사용자의 요청은 다음 흐름을 거친다:

User → HTTP → DNS → TCP → Server → Process → Port → Response

**중요한 질문**
"이 흐름이 동시에 1000명에게 발생하면?"

---

## 2. 로드 테스트의 본질

로드 테스트는 단순히 요청을 많이 보내는 것이 아니라  
**시스템이 어디서 무너지는지 찾는 과정이다**

---

## 3. 왜 로드 테스트가 필요한가?

### 단일 요청 vs 동시 요청

| 상황 | 결과 |
|------|------|
| 1명 | 정상 |
| 100명 | 느려짐 |
| 500명 | 장애 발생 |

**문제는 항상 "동시성"에서 발생한다**

---

## 4. 핵심 지표 (Metrics)

### 1) RPS (Requests Per Second)
- 초당 처리 가능한 요청 수

### 2) Latency
- 평균 응답 시간
- p95, p99 (느린 요청 기준)

### 3) Error Rate
- 실패한 요청 비율

### 4) Throughput
- 전체 처리량

---

## 5. 병목 발생 위치 (구조 기반 분석)

### 1) TCP 단계
- 연결 생성 비용 (3-way handshake)
- 최대 연결 수 제한

**증상: 연결 지연 증가**

---

### 2) 서버 (프로세스 / 스레드)
- 스레드 부족
- 이벤트 루프 블로킹

**증상: 요청 대기 증가**

---

### 3) 포트 / 커넥션
- 커넥션 풀 부족
- TIME_WAIT 증가

**증상: 연결 실패**

---

### 4) DB
- 느린 쿼리
- 락 (Lock)

**증상: 전체 응답 지연**

---

## 6. 로드 테스트 방법론

### Step 1. 가설 설정

예:
- "DB가 병목일 것이다"
- "CPU 사용률이 먼저 올라갈 것이다"

---

### Step 2. 점진적 부하 증가 (Ramp-up)

10 → 50 → 100 → 300 → 500 → 1000

---

### Step 3. 관찰 항목

- CPU 사용률
- 메모리
- 응답 시간
- 에러율

---

### Step 4. 한계 지점 기록

**"이 시점부터 성능이 급격히 나빠진다"**
